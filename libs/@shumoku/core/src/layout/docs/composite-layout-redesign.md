## 1. 背景：なぜ作り直すか

- **症状（test6 実測）**: 422リンク / 線幅 中央値14px・p75 34px・最大64px / 8px超が54% / キャンバス幅40%超を横断するリンク22本。太いリボンの束がノードを飲み込み、個々の経路が追えない。
- **根本原因（2つ）**:
  1. 配置エンジンが**グラフを「木1本」に還元**する（`primary parent`＝木親1つ、残りは overlay に退化）。test6 のような**非木の複合グラフ**で、捨てた構造が交差リボンになって戻る。
  2. **配線が実質ベジェのみ**（bus/detour は無効化、libavoid は PR #227 で削除）。422本を整理する仕組みが無い。
- **歴史的教訓**: 確立理論(ELK/dagre/libavoid)→限界→自作(flat-tree)→自作も限界。**共通の失敗＝単一の組織原理にコミットし、ネットワークの他次元をノイズ扱いした**こと。
- **今回の規律**: 「もっと賢いアルゴリズム」ではなく **複合性そのものを第一級の設計原理に据える**。エンジン選定はモデルが固まった後。

---

## 2. 設計原理

1. **複合因子を明示的な型／制約として先にモデル化する**（§3）。コードより先に紙の上で。
2. **因子が衝突したときの優先順位を確定する**。これが3回とも欠けていた工程。
3. **配置／配線／描画を3層分離し、エンジンをプラガブルに**。`ResolvedLayout` 契約を固定し、その裏を差し替える。
4. **既存の良い機構は捨てない**（§4）。作り直すのはコアの「木への還元」と空の配線だけ。
5. **test6 を受け入れテスト兼ベンチに固定**。「この絵が読めたら成功」を基準にする。

---

## 3. 複合因子モデル（本丸）

複合トポロジで2D平面を奪い合う因子。各々が「位置／描画にどう効くか」を制約として書く。

| 因子 | 位置への効き方（制約候補） | 現行の扱い | 再設計の方針 |
|---|---|---|---|
| 階層 (core/dist/access) | 有向分離制約（上流↑/下流↓） | flat-treeの木 | role駆動のtier + 有向分離 |
| グルーピング (subgraph) | **包含制約**（メンバを矩形内に） | hull＝後付け・配置を拘束しない | 配置を駆動する一級入力に |
| 容量 (帯域) | 原則レイアウト非駆動。描画(幅/色)へ | 線幅（暴走） | §7で幅から分離 |
| 役割 (device role) | tier割当の入力 | ほぼ未使用 | `role-tiers.ts`を駆動入力に |
| 多重接続/非木エッジ | 配線で束ねる（bundling） | overlayに退化→交差 | §6で本物の配線 |
| 状態/メトリクス | 描画overlay（色/太さ） | 既にあり(weathermap型) | 維持 |

**TODO: 因子衝突時の優先順位を確定する。** たたき台:
`包含(subgraph) > 非重なり > 階層(有向分離) > 容量短縮 > 交差最小化`
→ この順序が test6 で妥当かを検証してから確定。

## 4. アーキテクチャ（3層 + パイプライン）

```
(推定) → 配置(A) → 配線(B) → 描画/overlay(C)
         placement   routing    render
```

- **推定レイヤ**: Shumoku では discovery はプラグインが担当（Zabbix/NetBox/Prometheus）。**TODO: 推定を本再設計の範囲に含めるか**（基本は範囲外＝入力は確定済みグラフ、と仮置き）。
- **契約固定**: `ResolvedLayout`（nodes/ports/edges/subgraphs/bounds）を境界に。裏のエンジンを差し替えても描画は不変。
- **再利用する既存機構**（捨てない）:
  - `spacing.ts`（font metrics駆動の間隔導出）
  - `port-placement.ts`（ポート配置）
  - direction回転（TB/BT/LR/RL）
  - `pin.ts`（ピン留め）
  - hull描画（subgraph外接矩形の**描画**は残す。配置を拘束する用途に格上げ）
  - `diagnostics.ts`、既存テスト群

## 5. 配置コア（A）— 制約ベース化

「複合を同時に解く」唯一の理論系統は制約ベース（CoLa/VPSC, Cassowary）。包含・非重なり・有向分離・整列を**一つのソルバで同時**に満たす。

- **制約マッピング**:
  - subgraph → containment（包含）
  - 階層 → directed separation（有向分離）
  - role → tier 割当
  - 非重なり → hard non-overlap
  - 整列 → alignment（任意）
- **エンジン候補（TODO: 決定）**:
  - **A2. WebCola/VPSC 採用** — 包含・非重なり・分離が一級。複合性に理論的に最も近い。重さ/JS統合を検証。
  - **A2'. 自作軽量VPSCソルバ** — 依存を増やさず、Shumoku固有の制約だけ実装。
  - **A3. ELK layered** — 成熟。ただし subgraph=容れ物の摩擦（過去に不採用の理由）。再採用なら「前回なぜ合わなかったか」を明示的に潰す。
  - **A1. flat-tree 改修** — 交差最小化と多重接続を足す。最小工数だが単一原理の限界は残る。
- **判断軸**: 複合性の表現力 / 決定論性（現行の強み）/ 速度 / 依存とバンドルサイズ / 既存機構との接続。

## 6. 配線パス（B）— 本物のルーティング復活

400本超は配置だけでは救えない。配置と分離した独立パスにする。

- **方針**: bundling/nudging で横断リボンを束ね、**見た目はベジェを維持**（過去に直交はベジェ美学と衝突）。
- **候補（TODO: 決定）**:
  - Libavoid系 nudging（共有セグメントを交差を増やさず平行分離）の**発想を移植**
  - Holten 階層エッジバンドリング
  - 死蔵中の bus/detour を作り直すか破棄するか
- **要件**: source→target の対応が保たれること（旧bus が失敗した点）。

## 7. 容量表現（C）— 線幅から分離【即効・独立】

test6 の最大の犯人。レイアウトでなく**描画判断**。A/B と独立に先行できる。

- 線幅を **log/sqrt スケール + ハードキャップ**（〜6–8px）に。
- 帯域は **色／段階表示**（utilization legend の4段階のように）へ移す。
